# 業務要件ドキュメント

## 概要

このディレクトリには、OshiTalk（推しトーク）の業務要件定義書が含まれています。
技術仕様（`/docs/functional/functions/`）とは異なり、**ビジネスの視点**から各機能の目的、ユーザーストーリー、受け入れ条件を明確にしています。

## ドキュメント目的

- **非技術者（PO/PM/ビジネス）** でも理解できる要件定義
- **開発者** が実装すべき機能の「Why（なぜ）」と「What（何を）」を明確化
- **QA/テスター** が検証可能な受け入れ条件の提供
- **ステークホルダー** との認識合わせに使用

## ドキュメント構成

各業務要件ドキュメントは以下の構成で記載されています：

### 1. 概要
機能の目的とビジネス価値を簡潔に説明

### 2. ユーザーストーリー
```
As a [ユーザー種別],
I want to [行動],
So that [目的/ビジネス価値]
```

### 3. 機能要件（FR: Functional Requirements）
- 各要件にID付与（FR-001, FR-002...）
- 説明：何を実現するか
- 受け入れ条件：テスト可能な条件（チェックリスト形式）

### 4. 非機能要件（NFR: Non-Functional Requirements）
- パフォーマンス要件
- セキュリティ要件
- ユーザビリティ要件
- その他

### 5. 関連ファイル
- 実装ファイルへのリンク
- 技術仕様ドキュメントへのリンク

### 6. 変更履歴
- 日付、変更内容、理由を記録

## 技術仕様との関係

| ドキュメント種類 | パス | 対象読者 | 目的 |
|----------------|------|---------|------|
| **業務要件** | `/docs/requirements/` | PO/PM/ビジネス/QA | What/Why（何を、なぜ） |
| **技術仕様** | `/docs/functional/functions/` | 開発者 | How（どのように実装） |

業務要件は技術仕様を参照し、相互に補完し合う関係です。

## 使い方

### 新機能開発時
1. 該当する業務要件ドキュメントを確認
2. ユーザーストーリーとビジネス価値を理解
3. 受け入れ条件を確認
4. 技術仕様ドキュメント（`/docs/functional/functions/`）で実装詳細を確認

### 機能変更時
1. 業務要件ドキュメントを更新
2. 変更履歴に記録
3. 関連する技術仕様も更新

### テスト時
1. 受け入れ条件をテストケースとして使用
2. すべての条件を満たすことを確認

## 承認プロセス

業務要件の追加・変更は以下のプロセスに従います：

1. **提案**: 要件ドキュメントのドラフト作成
2. **レビュー**: PO/PMによるレビュー
3. **承認**: ステークホルダー承認
4. **実装**: 開発チームによる実装
5. **検証**: QAによる受け入れテスト
